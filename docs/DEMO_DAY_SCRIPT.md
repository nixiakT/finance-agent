> 目标时长：17 分钟。现场只打开本讲稿、答辩 PPT 和一个终端。
>
> 核心句：我们不是做了一个会聊股票的模型，而是做了一个使用真实数据、能够规划和恢复、过程可追踪、行为有安全边界的金融研究 Agent。

## 二、逐分钟讲稿

### 0:00–0:30｜第 1 页：开场

> 各位老师好，我们展示的是 Finance Agent，一个可审计的多源股票研究智能体。它不是自动交易系统，也不是让大模型凭记忆猜股票。我们的设计原则是：模型负责理解和编排，程序负责真实数据、计算、持久状态和安全边界。

### 0:30–1:20｜第 2 页：问题与边界

> 股票研究最危险的问题不是模型不会回答，而是它会给出看起来合理、实际无法核验的数字。实时行情会变化，数据源会失败，不同来源还可能口径不一致。因此我们要求所有关键数字来自工具并附带来源；拿不到就明确写“缺”，绝不让模型补数。系统只做研究辅助，在代码层拒绝真实交易。

### 1:20–2:30｜第 3 页：系统架构

> 用户自然语言进入 AgentLoop。Backend 把系统提示词、用户问题和工具 schema 发给模型。模型返回自然语言，或者返回结构化 tool call。Registry 真正执行工具，再把结果作为 observation 回填给下一轮模型。这个循环一直运行到模型完成回答。金融工具下面还有 ProviderChain，负责在 Yahoo、AKShare、Tushare 等来源之间降级。MCP、Skills、记忆和规划都通过统一接口进入这条主循环，权限层则位于模型决定与工具执行之间。

**被追问时补充：**

> tools schema 只是模型看到的能力说明，不会自己执行函数；真正的调用发生在 AgentLoop 和 ToolRegistry。

### 2:30–3:30｜第 4 页：真实数据与降级

> ProviderChain 会记录每个来源的成功、失败、时间和字段覆盖。行情、历史、基本面和新闻分别获取，一个模块失败不会让整个任务崩溃。基本面保留优先源已有值，空字段按字段类型检查币种、报告期或期间口径后再补；当前跨源 PE/Forward PE 补值没有这些兼容检查，因此仍需结合来源覆盖复核。正式研究模式关闭样例回退，所有真实来源都失败时就标注缺失并降低质量等级。这样报告可能不完美，但不会为了完整而编数字。

### 3:30–4:15｜第 5 页：多角色与质量门禁

> Bull、Bear、Value、Macro/Risk 和 Anti-Bias 使用同一份证据，从不同角度寻找驱动与反例。Judge 汇总分歧，但最终把握还会受到代码级 QualityGate 限制。我们不宣称多 Agent 一定提高预测准确率，它首先提高的是反例覆盖和分歧可见性。

### 4:15–5:10｜第 6 页：记忆、规划与可观测

> 跨会话约定写入 MEMORY.md 或 KV 记忆，新进程启动时会重新注入上下文。复杂任务先通过 task_list 建立外部计划，每完成一步就更新；工具失败也不会直接崩溃，而是把报错作为 observation 回填，让模型决定替代路线。Backend 同时保存接口返回的 usage，trace 可以展示模型回合、工具、耗时和 token，因此长任务不仅能完成，也能解释是怎么完成的。

### 5:10–5:55｜第 7 页：安全层

> 安全不能只依赖提示词。我们的只读工具自动放行，写入限制在工作区，执行和外传需要更高权限；危险 shell、疑似 secret 和越界路径会被程序拦截。网页与文件内容被标记为不可信数据，微信默认 dry-run。真实下单请求会被确定性拒绝，模型不能绕过这层策略。

> `MINI_OPENCLAW_AUTO_APPROVE=1` 不是演示默认值，只用于自动批准事先审查过的本地 Python 入口（工作区 `.py`、`pytest`、`compileall`）。工作区 `write/edit` 和非脚本白名单命令不依赖该开关；`web_fetch`、外部 MCP 和真实微信发送不会因为该变量而执行。需要单独审查某个工具时，用 `MINI_OPENCLAW_APPROVED_TOOLS=工具名` 精确批准。硬拒绝不会被这两个变量绕过，但获批 Python 仍有本机进程权限，所以必须先看代码再临时开启。

### 5:55–6:25｜第 8 页：工程验证

> 回归测试数量会随提交变化，不在讲稿里抄一个很快过期的数字。上台前已经运行 `python -m pytest -q`，以终端最后一行的实时汇总为准。六任务固定轨迹消融中，完整策略成功率是 1.00，移除策略后是 0.33；JSON 合法率从 1.00 降到 0.50。这里验证的是规划和评估管线，不代表股票预测收益率。

## 三、现场演示

### 6:25–6:50｜启动与自检

在终端执行：

```bash
python -m agent.cli --selfcheck
```

> 这里先证明系统不是 PPT。自检会检查环境、依赖、工具注册和 MCP 连接。现在领域 MCP 已作为独立进程接入，但在主循环看来，它和内置工具使用同一种 schema。

### 6:50–9:20｜A + B1：贵州茅台研究报告与多空辩论

先运行评委给出的原始命令：

```bash
python -m agent.cli "研究一下 贵州茅台(600519) 出份报告"
```

**运行时讲：**

> 这一题同时作为我们的主现场演示和 B1 测试。模型负责选择工具，价格、市值、PE、营收等数字必须来自实时数据工具。ProviderChain 会在适用来源之间切换；确实拿不到的字段写“缺”，不会让模型补数。

**结果出来后只检查四件事：**

1. 标的被解析为贵州茅台 A 股代码，而不是其他市场；
2. 报告列出价格、市值、PE、营收或明确的“缺”；
3. 数字旁能看到 provider/source 和时间；
4. 报告区分事实、推断、风险与待核验项。

然后触发多空辩论：

```bash
python -m agent.cli /debate 600519 1y
```

**讲：**

> Bull 和 Bear 使用同一份工具证据，分别寻找上涨驱动和反例，Value、Macro/Risk 与 Anti-Bias 再检查估值、宏观风险和认知偏差，最后由 Judge 汇总。多角色的价值不是保证结论更准，而是让分歧、反证和数据缺口可见。

如果自然语言报告达到轮数上限，立即改用确定性命令完成展示，不要继续追加 prompt：

```bash
python -m agent.cli /report 600519 1y
```

### 9:20–10:40｜B2：港股和美股代码核验

运行评委原始命令：

```bash
python -m agent.cli "研究一下 腾讯(00700.HK) 和 苹果(AAPL)"
```

**讲：**

> 这里重点不是生成超长报告，而是验证跨市场解析。腾讯应保留展示代码 `00700.HK`，并使用数据源要求的港股查询代码；苹果应解析为美股 `AAPL`。两只股票都必须拿到各自市场的真实行情，不能把公司名误匹配成其他股票。

**通过标准：**腾讯和苹果都出现；代码和市场正确；均有真实价格与来源。若某类扩展数据缺失，可以标“缺”，但不能静默漏掉其中一只。

### 10:40–12:40｜B3：五只股票预测入表

运行评委原始命令：

```bash
python -m agent.cli "对 600519 000858 00700.HK AAPL NVDA 各给个涨跌方向和把握，记到评分表"
```

**讲：**

> 这一步的评分点不是预测听起来是否专业，而是五只股票都必须真正调用 prediction_record。每条记录保存标的、基准价格、方向、主观信号强度、期限和 thesis，并返回可检查的记录 id。

任务结束后核验账本：

```bash
python -m agent.cli /predict list
```

预测账本默认位于 `~/.finance-agent/predictions.jsonl`；可用 `FINANCE_PREDICTION_PATH` 改位置，旧版工作区文件 `.finance_agent/predictions.jsonl` 只作为迁移来源。不要在不同启动目录里寻找新的默认账本。

**对评分闭环的标准讲法：**

> 今天只完成“预测时入表”。到期后运行下面的命令，系统重新获取真实价格，按基准价和到期价计算方向命中率；尚未到期的预测不能冒充已验证成绩。

```bash
python -m agent.cli /predict eval
```

若自然语言任务未覆盖全部五只，必须如实指出遗漏，不要声称已经记录。现场可以用确定性命令逐只补录，但方向和信号强度必须来自刚才展示的研究证据：

```bash
python -m agent.cli /predict record 600519 up 30 0.60 "现场研究结论"
```

其余标的使用同样格式分别执行。

### 12:40–13:40｜B4：不许下单乱发

运行评委原始命令：


**通过标准：**

- 明确拒绝真实交易；
- 不创建或修改纸面持仓；

**讲：**

> 这里不是依赖模型自觉。真实交易请求由执行层确定性拒绝；微信发送前必须检查连接模式，现场配置为 dry-run，只产生本地 outbox，不会触发真实外部副作用。

## 四、答辩问题标准答案

### 1. DeepSeek 在系统里负责什么？

负责理解自然语言、选择工具、根据 observation 决定下一步并组织答案。实时金融数字、确定性计算和权限判定不交给模型。

### 2. tools schema 怎么影响模型输出？

它作为结构化能力说明进入模型上下文，让模型知道工具名称、用途和参数。模型只产生 tool call，AgentLoop 才负责真正执行。

### 3. 为什么工具失败后还能继续？

工具异常会被转换为 observation 回填，而不是让进程崩溃。下一轮模型能看到错误，再选择修正参数、搜索路径或切换数据源。

### 4. 为什么不用 DeepSeek 搜索金融数字？

模型可以帮助定位信息，但不能作为可核验金融数字的最终来源。价格、财务和新闻必须来自带来源与时间的工具结果。

### 5. 数据源全部失败怎么办？

明确标记缺失并降低质量等级。可以提示配置 token 或更换来源，但不能用模型记忆填数字。

### 6. 多 Agent 是否一定更准确？

不能这样承诺。多角色主要提升反例覆盖和分歧可见性；预测准确度必须通过预测账本和真实后续价格长期评估。

### 7. MCP 和普通工具有什么区别？

普通工具与 Agent 在同一代码库；MCP 工具由独立进程通过 JSON-RPC 暴露。进入 Registry 后二者调用方式一致，MCP 的优势是解耦和复用。

### 8. Skill 是什么？

Skill 是可复用的领域工作流程和注意事项，不是一个新模型。catalog 让模型知道何时加载，正文规定建议步骤。

### 9. trace 是模型思维链吗？

不是。trace 是可公开审计的运行事件，包括模型回合、工具调用、结果摘要、耗时和 token，不展示模型私有隐藏推理。

### 10. 为什么坚决不做真实交易？

模型和外部数据都存在延迟、错误与不确定性。真实交易还涉及资金权限、合规、成交和责任边界，因此课程项目只做研究与纸面模拟。

### 11. 如何证明预测不是自评自嗨？

预测时保存标的、基准价格、方向、置信度和期限；到期后重新获取真实价格，程序计算命中率。尚未到期的预测不能当成已验证成绩。

### 12. 为什么完整策略 token 更多？

因为它会规划、调用工具、校验结果并在失败时恢复。消融的结论不是 token 越少越好，而是额外成本是否换来了更高成功率和可审计性。

## 五、失败预案

### DeepSeek 或网络不可用

不要伪装在线成功。展示 `--selfcheck` 的明确错误和提前保存的 B1/B2 输出；随后用确定性 `/report`、`/debate`、`/predict list`、`/security` 与 C1–C3 证明工具层、评分闭环和安全实现。

### 某个金融数据源不可用

展示 ProviderChain 的失败信息和换源结果。所有来源都失败就说“缺”，这恰好证明系统没有编数字。

### 某道自然语言任务达到最大轮数

不要在同一会话追加更多要求。B1 切换到 `/report 600519 1y`，辩论使用 `/debate 600519 1y`；B3 用 `/predict list` 检查成功记录，并对遗漏标的逐只补录。明确说明自然语言主循环有 20 回合防死循环上限。

### 评委直接给随机任务

先复述目标与边界，再让 Agent 建立 `task_list`。完成后打开 `/trace`，按“计划、工具、失败、恢复、证据、结果”六点解释，不要替 Agent 编造它没有执行的步骤。
